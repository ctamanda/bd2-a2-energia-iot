# Q4 - Predicado não-sargable em data

## A consulta

```sql
SELECT COUNT(*)
FROM energia.leituras
WHERE DATE(occurred_at) = DATE '2026-05-09';
```

## Diagnóstico

Esta consulta conta os chamados abertos em uma data específica. O problema está no
uso da função `DATE()` diretamente sobre a coluna `occurred_at`: ao envolver a
coluna em uma função, o predicado torna-se **não-sargable** — o índice B-tree em
`occurred_at` não pode fazer uma varredura de intervalo, porque ele armazena os
timestamps originais, não os valores transformados por `DATE()`. O resultado é
Seq Scan completo nas 250 mil linhas da tabela `leituras`.

**Plano antes (sem índice, predicado não-sargable):**
```
Finalize Aggregate
  -> Gather
       Workers Planned: 2 / Workers Launched: 2
       -> Partial Aggregate
            -> Parallel Seq Scan on leituras
                 Filter: (date(occurred_at) = '2026-05-09'::date)
                 Rows Removed by Filter: 82.897/worker
                 Buffers: shared hit=6931 read=995
Execution Time: 27.830 ms
```

## Hipótese

Se reescrevermos o filtro como uma faixa de timestamps explícita
(`occurred_at >= '2026-05-09' AND occurred_at < '2026-05-10'`), a coluna aparece
**nua** no `WHERE`, sem função envolvente. O predicado torna-se **sargable** e o
banco pode fazer um Index Only Scan seletivo, lendo apenas as entradas do índice
que caem dentro da faixa do dia alvo.

## Otimização aplicada

Criação de índice B-tree em `occurred_at` e reescrita do predicado como faixa:

**Índice criado:**
```sql
CREATE INDEX idx_leituras_occurred
    ON energia.leituras (occurred_at);
ANALYZE energia.leituras;
```

**Consulta reescrita (sargable):**
```sql
SELECT COUNT(*)
FROM energia.leituras
WHERE occurred_at >= '2026-05-09 00:00:00'
  AND occurred_at < '2026-05-10 00:00:00';
```

## Resultado

- **Antes:** Parallel Seq Scan em 250 mil linhas, **28 ms**, 995 blocos lidos do
  disco.
- **Depois:** Index Only Scan com range condition diretamente no índice, **0,67 ms**
  e apenas **6 buffers** — acesso direto às 1.309 linhas do dia alvo, sem ler
  nenhum bloco de heap.
- **Ganho:** aproximadamente **42× mais rápido**, leituras de disco eliminadas.

A semântica é idêntica: a faixa `>= dia 00:00:00 AND < dia+1 00:00:00` captura
todos os timestamps do dia inteiro, inclusive `23:59:59.999`.

## Custo Colateral

- O índice ocupa **5,4 MB** em disco.
- A consulta reescrita é mais verbosa, mas pode ser parametrizada com dois
  bind-parameters na camada de aplicação.
- O índice em `occurred_at` é reutilizável por outras consultas de intervalo
  temporal na tabela `leituras` — também beneficia a Q3.

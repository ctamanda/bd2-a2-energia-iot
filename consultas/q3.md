# Q3 - Filtro em JSONB

## A consulta

```sql
SELECT id, titulo, tags, occurred_at
FROM energia.leituras
WHERE tags @> '{"produto":"app-mobile"}'
ORDER BY occurred_at DESC
LIMIT 80;
```

## Diagnóstico

Esta consulta localiza chamados associados a um produto específico, filtrando pelo
campo `tags` do tipo JSONB com o operador de contenção (`@>`). Sem índice sobre
`tags`, o banco é forçado a desserializar e comparar o documento JSON de cada uma
das 250 mil linhas — resultando em Parallel Seq Scan com leitura de blocos do
disco.

**Plano antes (sem índices de otimização):**
```
Gather Merge
  Workers Planned: 2 / Workers Launched: 2
  -> Parallel Seq Scan on leituras
       Filter: (tags @> '{"produto": "app-mobile"}')
       Rows Removed by Filter: 69.444/worker
       Buffers: shared hit=6091 read=1909
Execution Time: 43.443 ms
```

## Hipótese

O operador `@>` é nativamente suportado por índices GIN no PostgreSQL. Um índice
GIN sobre `tags` indexa cada chave e valor do documento JSONB individualmente,
dando ao otimizador a opção de localizar diretamente as linhas que contêm o par
pesquisado. Com o índice GIN criado em conjunto com o `idx_leituras_occurred`
(B-tree em `occurred_at`, criado na Q4), o planner escolhe o plano mais eficiente
para a combinação de `ORDER BY occurred_at DESC` e `LIMIT 80`.

## Otimização aplicada

Criamos um índice GIN na coluna `tags`:

```sql
CREATE INDEX idx_leituras_tags_gin
    ON energia.leituras USING GIN (tags);
ANALYZE energia.leituras;
```

## Resultado

- **Antes:** Parallel Seq Scan em 250 mil linhas, **43 ms**, 1.909 blocos lidos
  do disco.
- **Depois:** Index Scan Backward via `idx_leituras_occurred` (B-tree de
  `occurred_at`), verificando apenas ~412 linhas até encontrar as 80 mais recentes
  com o produto correto — **0,66 ms**, zero blocos lidos do disco.
- **Ganho:** aproximadamente **65× mais rápido**, leituras de disco eliminadas.

> **Nota sobre a escolha do plano:** para `ORDER BY occurred_at DESC LIMIT 80`
> com ~1/6 das linhas satisfazendo o filtro JSONB, o planner prefere o scan
> backward no B-tree (entrega a ordenação de graça e para no 80º resultado) ao
> GIN + Bitmap Heap Scan (que buscaria as 41 mil linhas correspondentes e depois
> ordenaria). O GIN seria escolhido para consultas sem `ORDER BY occurred_at` ou
> com `LIMIT` alto.

## Custo Colateral

- O índice GIN ocupa **1,4 MB** em disco.
- Escritas (INSERT/UPDATE) na coluna `tags` ficam mais lentas, pois o GIN precisa
  indexar cada entrada do documento JSONB separadamente.
- O índice habilita acesso eficiente para `@>`, `?`, `?|` e `?&` sobre `tags`,
  sendo reutilizável para buscas por produto, categoria, urgência etc.

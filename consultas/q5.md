# Q5 - DISTINCT mascarando explosão de JOIN

## A consulta

```sql
SELECT DISTINCT e.id, e.nome, e.segmento
FROM energia.unidades e
JOIN energia.leituras r ON r.unidades_id = e.id
JOIN energia.eventos_leitura ev ON ev.leituras_id = r.id
WHERE r.status = 'concluido'
  AND ev.occurred_at >= NOW() - INTERVAL '30 days';
```

## Diagnóstico

Esta consulta busca as unidades que tiveram pelo menos um chamado concluído com
evento nos últimos 30 dias. O problema está no caminho de execução: o duplo JOIN
multiplica as linhas — cada unidade aparece tantas vezes quantas forem as
combinações entre seus chamados e os eventos desses chamados. O `DISTINCT` só
elimina os duplicados **depois** de gerar e ordenar todo esse volume, forçando um
`HashAggregate` custoso no topo do plano.

**Plano antes (sem índices de otimização):**
```
HashAggregate  (DISTINCT — acumula 25.330 linhas antes de devolver 9.565)
  -> Gather
       Workers Planned: 2 / Workers Launched: 2
       -> Hash Join
            -> Parallel Hash Join
                 -> Parallel Seq Scan on eventos_leitura ev
                      Filter: (occurred_at >= now() - '30 days')
                      Rows Removed by Filter: 219.912/worker
                 -> Parallel Seq Scan on leituras r
                      Filter: (status = 'concluido')
                      Rows Removed by Filter: 66.667/worker
            -> Hash (unidades)
  Buffers: shared hit=14211 read=13921
Execution Time: 455.010 ms
```

## Hipótese

O `DISTINCT` aqui é sintoma de que a intenção real é **"existe ao menos um evento
que satisfaça as condições?"**. Substituir os JOINs por `EXISTS` transmite essa
semântica diretamente ao otimizador: ele converte para um **semi-join** que para
imediatamente ao encontrar a primeira linha válida para cada unidade, sem gerar o
produto cartesiano. Índices compostos em `leituras(unidades_id, status)` e em
`eventos_leitura(leituras_id, occurred_at)` eliminam os Seq Scans nas subtabelas.

## Otimização aplicada

Reescrita com `EXISTS` e índices de suporte:

**Índices criados:**
```sql
CREATE INDEX idx_leituras_unidades_status
    ON energia.leituras (unidades_id, status);

CREATE INDEX idx_eventos_leituras_occurred
    ON energia.eventos_leitura (leituras_id, occurred_at);

ANALYZE energia.leituras;
ANALYZE energia.eventos_leitura;
```

**Consulta reescrita:**
```sql
SELECT e.id, e.nome, e.segmento
FROM energia.unidades e
WHERE EXISTS (
    SELECT 1
    FROM energia.leituras r
    JOIN energia.eventos_leitura ev ON ev.leituras_id = r.id
    WHERE r.unidades_id = e.id
      AND r.status = 'concluido'
      AND ev.occurred_at >= NOW() - INTERVAL '30 days'
);
```

## Resultado

- **Antes:** HashAggregate + Parallel Seq Scans, **455 ms**, 13.921 blocos lidos
  do disco.
- **Depois:** Hash Right Semi Join com `idx_eventos_leituras_occurred`, **157 ms**,
  2.881 blocos lidos — o semi-join interrompe a busca ao primeiro match por
  unidade, sem HashAggregate.
- **Ganho:** aproximadamente **3× mais rápido**; leituras de disco caem de
  13.921 para 2.881 blocos (redução de 79%).

**Plano depois:**
```
Hash Right Semi Join
  Hash Cond: (r.unidades_id = e.id)
  -> Gather
       -> Nested Loop
            -> Parallel Seq Scan on leituras r  (filter status='concluido')
            -> Index Only Scan using idx_eventos_leituras_occurred
                 Index Cond: (leituras_id = r.id AND occurred_at >= now()-30d)
                 Index Searches: 50.000
  -> Hash (unidades)
Execution Time: 157.126 ms
```

## Custo Colateral

- `idx_eventos_leituras_occurred` ocupa **23 MB** em disco.
- `idx_leituras_unidades_status` ocupa **3,7 MB** em disco.
- INSERTs em `eventos_leitura` e `leituras` ficam ligeiramente mais lentos.
- A consulta reescrita preserva a semântica original: retorna exatamente as mesmas
  9.565 unidades, sem duplicatas, sem montar o produto cartesiano.
- O ganho seria mais expressivo em bases com mais eventos recentes ou com
  `work_mem` menor (onde o HashAggregate precisaria usar disco temporário).

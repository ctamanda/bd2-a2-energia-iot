# Q2 - Busca por e-mail ignorando maiúsculas/minúsculas

## A consulta

```sql
SELECT id, nome, email, cidade, segmento
FROM energia.unidades
WHERE UPPER(email) = UPPER('contato410@empresa410.com.br');
```

## Diagnóstico

Esta consulta busca uma unidade pelo e-mail digitado, sem diferenciar
maiúsculas de minúsculas. A coluna `email` já possui um índice `UNIQUE`, mas ele
não é aproveitado: ao aplicar a função `UPPER()` sobre a coluna, o banco não
consegue mais usar o índice existente e cai em Seq Scan.

Na prática, o banco lê as 50 mil linhas da tabela, aplica `UPPER()` em cada
e-mail e descarta quase todas para devolver apenas uma.

## Hipótese

Como a aplicação sempre compara `UPPER(email)`, acreditamos que um índice de
expressão sobre `UPPER(email)` armazene exatamente o valor pesquisado,
permitindo um acesso direto à linha sem varrer a tabela.

## Otimização aplicada

Criamos um índice de expressão sobre `UPPER(email)`:

```sql
CREATE INDEX idx_unidades_upper_email
    ON energia.unidades (UPPER(email));
ANALYZE energia.unidades;
```

Consideramos também comparar `email` diretamente, o que usaria o índice
`UNIQUE` já existente. Descartamos essa opção porque mudaria o comportamento da
busca, passando a diferenciar maiúsculas de minúsculas.

## Resultado

- **Antes:** o banco usava Seq Scan e levava cerca de 88 ms, lendo 797 blocos.
- **Depois:** passou a usar Index Scan, caindo para cerca de 0,3 ms e apenas
  4 blocos.
- **Ganho:** a consulta ficou praticamente instantânea, com acesso direto a uma
  única linha.

## Custo Colateral

- O índice ocupa cerca de 2,4 MB em disco.
- Cada INSERT ou UPDATE de e-mail passa a recalcular `UPPER()` e atualizar o
  índice, mas como `unidades` é uma tabela cadastral com pouca escrita, esse
  custo é baixo.
- O índice só é usado se a consulta escrever exatamente `UPPER(email)`. Vale
  padronizar essa forma na aplicação para garantir o aproveitamento.
